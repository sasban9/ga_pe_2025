Under the hood

How Vector Search Works

Documents compressed into high-dimensional vectors. Similarity = geometric proximity. No semantic structure preserved.

The cracks

Where RAG 1.0 Breaks

  • ⚡
    Semantic gap — embedding similarity ≠ logical relevance. "Capital of France" may miss "Paris is located in…"
  • 🔗
    No relationships — chunks are atomic; multi-hop reasoning requires following entity links the DB doesn't have.
  • 📄
    Chunking artifacts — splitting documents at fixed tokens destroys cross-sentence context.
  • 🏷️
    No metadata awareness — recency, authority, and source hierarchy are invisible to cosine distance.
  • 🔄
    Poor multi-step queries — "Who founded the company acquired by Google in 2014?" requires chained lookups.
Evolution

1.0 → 2.0: Hybrid Retrieval

RAG 3.0  ·  2024–2025

Enter the Knowledge Graph

RAG 3.0 replaces flat vector stores with structured knowledge graphs — entities and relationships stored as typed triples, enabling multi-hop traversal, logical inference, and dynamic context construction.


(Entity A)  —[RELATION]→  (Entity B)
Structure

Knowledge Graph Anatomy

Typed nodes + directed labeled edges. Every fact is a traversable relationship, not a floating embedding.

Data model

The Triple is the Atom

// Subject → Predicate → Object

(OpenAI) FOUNDED_BY (Sam Altman)
(OpenAI) LOCATED_IN (San Francisco)
(OpenAI) PRODUCES (GPT-4)
(GPT-4) IS_A (LLM)
(GPT-4) RELEASED_ON (2023-03-14)

// Multi-hop: Who founded the company that makes GPT-4?
MATCH (?company)-[PRODUCES]→(GPT-4)
      MATCH (?company)-[FOUNDED_BY]→(?person)
RETURN ?person  // → Sam Altman

How to build it

Graph Construction Pipeline

1 · Document Ingestion

Parse PDFs, HTML, databases. Sentence segmentation. Coreference resolution.

2 · Entity & Relation Extraction

NER + RE models (SpanBERT, REBEL) or prompt LLM for triplet extraction.

3 · Graph Canonicalization

Entity linking, deduplication via ontology. Store in Neo4j / RDF triple store.

4 · Hybrid Index

Embed node summaries → vector index for fuzzy entry. Graph for precise traversal.

Retrieval modes

RAG 3.0 Query Strategies

Graph Traversal
  • Cypher / SPARQL queries
  • Anchor on extracted entities
  • Follow typed edges N hops
  • Collect subgraph context
Semantic Fallback
  • ANN search on node embeddings
  • Handles paraphrase & typos
  • Fuzzy entity disambiguation
  • Blended ranking with graph score

Best results: graph for known facts, vectors for unknown territories.

Honest reckoning

RAG 3.0 Challenges

Hard problems
  • Graph construction is expensive
  • Extraction errors compound
  • Schema design is domain-expert work
  • Graph freshness / update latency
  • Cypher skill gap for most teams
Mitigations
  • LLM auto-extraction pipelines
  • Confidence scores per triple
  • Auto-schema induction from docs
  • Incremental graph update streams
  • NL→Cypher translation layers
Decision framework

Choosing Your RAG Tier

  • 1.0
    Use RAG 1.0 when — simple Q&A over homogeneous docs, quick prototype, <1M chunks, no multi-hop needed.
  • 2.0
    Use RAG 2.0 when — mixed content types, precision matters, BM25 handles keyword-heavy queries, re-ranking improves answers.
  • 3.0
    Use RAG 3.0 when — entities and relationships are central, multi-hop queries, need provenance & explainability, enterprise knowledge bases.
What's next
